## 4. PoCモデル：今あるデータで作れるもの

**目的：**「この地域・このサービス・この時期に手配難が起きそうか」を市区町村単位でスコアリングする。

### 使う特徴量

```
需要側：
  - 過去N週の市区町村 × サービス種別の受付件数
  - 月・曜日・季節性
  - 過去の手配難発生率（同エリア・同サービス）

供給側：
  - 稼働業者数（市区町村 × サービス種別）
  - 評価スコア（品質プロキシ）
  - 24時間対応可否
```

### 目的変数

作業打診未完了 の発生（引野さん・羽星さんと定義を合意した上で確定）

### アウトプットイメージ

```
市区町村         | サービス | 週       | リスクスコア | リスクランク
[23203] 愛知県一宮市 | 水道   | 4月第1週 |    0.78    | 高（赤）
[13211] 東京都小平市 | 電気   | 4月第1週 |    0.31    | 低（緑）
```

→ Exploratoryで地図上に色付け表示（可視化はJBR側で実装。ロックインなし）

### 精度の期待値

- 2.5年分のトランザクションで季節性・地域性のパターンは十分学習できる
- 業者の実質能力が不明なため供給側の精度に限界あり
- まずはルールベーススコアリング → 精度検証 → MLモデル（LightGBM等）へ移行

## 5. データ構造を整えると何が変わるか

### Phase 1 ｜ 今すぐ：既存データでPoC

```
できること：
  ・地域 × サービス × 月次の手配難リスクマップ
  ・供給過多エリアの特定（営業強化の根拠）
  ・社内合意を取るための「見える化」

限界：
  ・業者の実力が見えないため精度に上限あり
  ・「誰をアサインすべきか」はわからない
```

### Phase 2 ｜ データ基盤整備後：精度が上がる

```
やること：
  ・マスタの一元管理
  ・フラット構造の正規化・ETL高速化
  ・業者ケイパビリティ情報の収集（評価加点インセンティブ）

追加でできること：
  ・業者の実質処理能力を加味したスコアリング
  ・「この地域は3社いるが実質1社分の供給力」が見える
  ・基幹システムへの依存度が下がり、カラム追加コストが激減
```

### Phase 3 ｜ トランザクションログ活用後：競合優位へ

```
やること：
  ・作業開始〜完了のリードタイム記録
  ・業者ごとの実績（到着時間・解決率・クレーム率）の蓄積
  ・通話ログの構造化

できること：
  ・「電話が来た瞬間に誰を送るか」のリアルタイム最適化
  ・業者がデータを入れると仕事が来る仕組み（入力インセンティブの自動化）
  ・フィードバックループによる継続的な精度改善
```
