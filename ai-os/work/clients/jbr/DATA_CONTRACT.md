## 3. データアセスメント：見つかった問題点

### ① 464カラムのフラット構造

本来は「受付」「作業依頼」「サービス書」「業者情報」として別テーブルで管理されるべきデータが、1行にすべて詰め込まれている。

```
現状：受付1件 = 1行 × 464カラム（大半が空欄）
理想：受付テーブル ＋ 作業依頼テーブル ＋ サービス書テーブル（正規化）
```

**影響：** 処理が重い / 集計しにくい / カラム追加に数ヶ月・高コスト

---

### ② マスタの分散・非連動

業者情報が3箇所に分かれており、それぞれが独立して管理されている。

| 場所 | 内容 |
|------|------|
| Googleスプレッドシート | 対応エリア・サービス種別・評価値 |
| 基幹システム | 契約情報・請求情報 |
| 対応エリアCSV | 市区町村単位の対応可否 |

**影響：** 片方を更新しても他に反映されない。整合性が保てない。

---

### ③ 手配難の定義が統一されていない

上記のとおり3つのシグナルが存在するが、「どれが手配難か」の定義が社内でも統一されていない。

**影響：** 何を予測するモデルを作るかが確定できない。定義の合意が先決。

---

### ④ 業者のケイパビリティ情報が欠落

「水道対応可」はわかるが、「何人いるか」「資格は何か」「1日何件対応できるか」が管理されていない。

**影響：** エリアに業者が3社いても実質的な供給量が見えない。モデルの精度に上限が生じる。

---

### まとめ：今すぐ使えるもの / 使えないもの

| データ | 状態 |
|--------|------|
| 市区町村単位の需要（受付件数） | ◎ 使える |
| 市区町村単位の供給（稼働業者数・エリア） | ◎ 使える |
| サービス種別の分類 | ◎ 使える |
| 手配難の発生実績（フラグ） | ○ 使える（定義の合意が必要） |
| 業者の実質的な処理能力 | × データなし |
| 業者の移動時間・到着実績 | × データなし |
| 通話・チャットログ | × 未構造化 |

## 手配難シグナルの分布（2026年2月分）

データ内に手配難を示すシグナルが3層存在することが確認できた。

| シグナル | 発生件数 | 意味 |
|----------|---------|------|
| 手配有無フラグ = 手配あり | 全体の約45% | 業者への手配が発生した |
| 作業打診未完了 | 全体の約45% | 打診したが決まらなかった（手配難の予兆） |
| 作業依頼未完了 | 全体の約37% | 依頼を出したが受注されなかった |
| クレーム「手配困難」 | 175件 / 4ヶ月 | 実害として顕在化した手配難 |
