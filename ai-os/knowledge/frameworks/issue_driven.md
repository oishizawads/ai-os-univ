# イシューからはじめよ — 思考フレームワーク

> 出典: 安宅和人『イシューからはじめよ』  
> 用途: あらゆる思考・分析・判断のベースフレームとして使う

---

## 0. 根本原則

```
Value = イシュー度 × 解の質
```

- **イシュー度**: その問いに答えることが、どれだけ意味を持つか
- **解の質**: その問いにどれだけ正確・深く答えられているか
- どちらかが低ければ、もう片方をどれだけ上げても価値はゼロに近い

### 犬の道（避けるべきパターン）

「とにかく手を動かす」「量で解の質を上げる」→ イシュー度が低ければ消耗するだけ。

正しい順序は **イシュー度を先に担保する**。

---

## Step 1: イシューを見極める

### 良いイシューの3条件

| 条件 | 問い |
|------|------|
| **本質的な選択肢を含む** | 答えによって、行動・判断が変わるか？ |
| **深い仮説がある** | 常識を覆すか、新しい構造で世界が見えるか？ |
| **答えを出せる** | 現実的に検証・判断できるか？ |

3つ揃わなければ、そのイシューは「やるべき問い」ではない。

### イシューを特定するための問いかけ

```
1. 自分は今、何を明らかにしようとしているか？
2. その答えが出たとして、何が変わるか？
3. 答えが「AかBか」で意味のある差が生まれるか？
4. 今の自分に答えを出す手段はあるか？
```

### よくある罠

- **問いが広すぎる**: 「〇〇はどうあるべきか」→ 答えようがない
- **結論が決まっている**: 仮説ではなく確認作業になっている
- **答えても何も変わらない問い**: 知的好奇心の充足にすぎない

---

## Step 2: 仮説を立てる（問い → スタンス）

問いを「〜ではないか」という **仮説（スタンス）** に変換する。

```
問い:  「なぜ精度が上がらないのか？」
仮説:  「ラベルノイズではなく、特徴量の分布シフトが主因ではないか」
```

仮説があると：
- 分析の焦点が絞られる
- 何を見れば「仮説が崩れるか」が明確になる
- 無駄な分析をしない

### スタンスの質を上げる問い

```
- この仮説が正しければ、何が説明できるか？
- この仮説が間違っていれば、何が観察されるはずか？
- 今まで「当たり前」とされてきた前提を疑っているか？
```

---

## Step 3: イシューを分解する

### 分解の原則: MECE

- **Mutually Exclusive**: 重複なし（同じことを二度考えない）
- **Collectively Exhaustive**: 漏れなし（重要な観点を落とさない）

### 分解パターン

| パターン | 使いどき |
|---------|---------|
| **プロセス分解** | 原因が流れの中のどこかにある |
| **構造分解** | 全体を構成要素に分ける |
| **軸で切る** | 2軸マトリクスで象限に分ける |

### ストーリーラインの構造

分解したサブイシューを **So what（だから何？）** でつないで、一本の論理にする。

```
サブイシューA → [So what] →
サブイシューB → [So what] →
サブイシューC → [So what] → 最終イシューへの答え
```

- **縦の論理**: 上位イシューと下位イシューが因果でつながっている
- **横の論理**: 同じ階層のサブイシューが MECE になっている

---

## Step 4: 解を出す（分析設計 → 実行）

### 分析の3基本パターン

| パターン | 問い |
|---------|------|
| **比較** | AとBでどちらが大きいか・違うか |
| **構成** | 全体のうち何がどの程度を占めるか |
| **変化** | 時間やパラメータによってどう変わるか |

### 分析設計の順序

```
1. アウトプットイメージを先に描く（何が見えれば仮説を検証できるか）
2. そのために必要なデータ・手法を決める
3. 実行する
```

**「データがあるから分析する」ではなく「仮説を検証するために必要なものを揃える」**

### 仮説検証の問い

```
- この分析結果は、仮説を支持するか？否定するか？
- 反証となるデータを積極的に探したか？
- 「見たいものしか見ていない」確証バイアスに陥っていないか？
```

---

## Step 5: 伝える

### メッセージファーストの原則

結論から先に言う。理由・証拠は後。

```
[悪い例] データを分析した結果、〇〇であることが分かりました。
[良い例] 〇〇です。なぜなら…
```

### ストーリーラインを磨く問い

```
- イシューへの答えが一言で言えるか？
- 各サブイシューの答えが、上位イシューの答えを支えているか？
- 聞き手が「だから何？」と思う部分はないか？
```

---

## 実践チェックリスト

### 何かを始める前に

```
[ ] 今解こうとしている問いは何か？（一文で書けるか）
[ ] その答えが出たとして、何が変わるか？
[ ] 仮説（スタンス）はあるか？
[ ] イシューをサブイシューに分解できているか？
```

### 作業中に

```
[ ] 今の作業はどのサブイシューに紐づいているか？
[ ] 仮説を否定するデータを探したか？
[ ] 犬の道に入っていないか（量で押し切ろうとしていないか）？
```

### まとめる前に

```
[ ] 最終イシューへの答えを一文で言えるか？
[ ] So what の連鎖が成立しているか？
[ ] 聞き手の行動・判断は変わるか？
```

## イシュー度の判断マトリクス

```
                  高
                   │  やるべき       本命
    イシュー度      │  （でも急ぐな）  （今すぐやる）
                   │
                   │  やらない       時間の無駄
                   │  （捨てる）     （犬の道）
                  低└──────────────────────────
                     低              高
                          解の質への投資コスト（難易度）
```

コスト低 × イシュー度高 → **即着手**  
コスト高 × イシュー度低 → **即捨てる**  
コスト高 × イシュー度高 → **慎重に設計してから取り組む**

## Claude に適用する場合のプロンプト冒頭

```
イシュー駆動で考えてください。
1. まず「何を明らかにすべき問いか」を特定する
2. 仮説（スタンス）を立てる
3. MECEに分解する
4. 各サブイシューを検証する
5. So what でまとめる
---
[本題をここに記述]
```
