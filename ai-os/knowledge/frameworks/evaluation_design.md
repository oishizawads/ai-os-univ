## Summary
評価指標の設計はモデル設計より先に行う。
「精度が上がった」は成功ではない。「ビジネス指標に繋がる指標が上がった」が成功。
Offline評価とOnline評価のギャップを事前に想定する。

## 評価指標の選び方

### ビジネス指標 → ML指標の変換
| ビジネス指標 | ML指標候補 | 注意点 |
|---|---|---|
| 売上最大化 | RMSE, MAE（売上予測） | スケールに注意。対数変換で安定することが多い |
| 不良品ゼロ | Recall（異常検知） | Precisionとのトレードオフを設計する |
| 離脱率低下 | AUC, LogLoss（離脱予測） | スコアリングの閾値で現場実装が変わる |
| 推薦精度 | NDCG, MAP, Recall@K | K（上位何件か）をビジネス要件から決める |
| 時間コスト削減 | MAPE, sMAPE（時系列） | ゼロ近くの値でMAPEが爆発するので注意 |

### 指標選択の基準
1. **感度**: 良い予測と悪い予測を区別できるか
2. **安定性**: データの偏りや外れ値に過度に影響されないか
3. **解釈可能性**: 関係者（非技術者含む）が理解できるか
4. **最適化可能性**: モデルが直接最適化できる（またはSurrogateが使える）か

## Offline / Online ギャップの設計

### ギャップが発生する主な原因
- **分布シフト**: 学習データと本番データの分布が違う
- **フィードバックループ**: モデルの予測が次の入力に影響する（推薦システム等）
- **Population drift**: ユーザー行動・市場環境の変化
- **Implementation gap**: OfflineのバッチとOnlineのリアルタイムで前処理が違う

### 対処策
- Offline evalにはなるべく本番に近い時間分割を使う
- Shadow deployで本番データでOffline評価を先に行う
- A/Bテストの設計を事前に決めておく（何を比較するか・サンプルサイズ）

## 代理指標（Proxy Metric）の設計

直接測れないものをどう評価するか。

### よい代理指標の条件
- 最終指標と相関がある（因果的に繋がっている）
- 短期間で計測できる
- 操作しにくい（ゲーミングされにくい）

### 代理指標の落とし穴
- 代理指標を最適化したら最終指標が下がった（Goodhart's Law）
- 複数の代理指標が矛盾する
- 代理指標だけ見ていてビジネス文脈を忘れる

## 多指標の優先順位付け

複数の指標がある場合:
1. **Guard metric**（下回ったら問題）: 精度 > 0.8、Latency < 100ms 等
2. **Primary metric**（最大化する主目標）: 1つに絞る
3. **Secondary metric**（参考値）: 意思決定に使うが主目標ではない

複数を同時に最大化しようとしない。主目標を1つ決める。

## 評価設計のドキュメント化

```
評価指標: （名前と計算式）
選んだ理由: （ビジネス目標との接続）
Guard metric: （これを下回ったら失格）
計測方法: （Offline / Shadow / A/B）
成功条件: （現状比〇〇%改善）
失敗条件: （これが出たら方針変更）
Offlineとのギャップリスク: （想定される乖離と対処）
```

## Anti-patterns
- 最適化しやすいからという理由でAccuracyを使う（不均衡データで無意味）
- 複数指標を同時に最大化しようとする
- Offline evalだけで「良い」と判断してリリースする
- ビジネス指標との接続を確認せずにML指標だけで議論する

## Eval
- ビジネス目標とML指標が文書で繋がっているか
- Guard metricが定義されているか
- Offline/Onlineギャップのリスクを考慮したか
- 成功・失敗条件が数値で定義されているか
