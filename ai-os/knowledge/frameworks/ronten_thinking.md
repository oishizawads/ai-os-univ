# 論点思考 — 思考フレームワーク

> 出典: 内田和成『論点思考 BCG流 問題設定の技術』（東洋経済新報社）  
> 著者: 内田和成（ボストン コンサルティング グループ 日本代表）  
> 関係: 『仮説思考』の前段。「問題を解く前に、正しい問題を設定する」技術  
> 用途: あらゆる思考の最初のステップ。「何を考えるか」を決める

---

## 0. 根本原則

```
間違った問題を正しく解いても、価値はゼロどころかマイナスになる。
論点思考とは「解くべき問題を正しく設定する」技術である。
```

仮説思考との関係:
- **論点思考**: 「何を問題にすべきか」を決める（問題設定）
- **仮説思考**: 設定した問題に対して「仮説」を立てて解く（問題解決）

この2つはセットで機能する。論点思考なしに仮説思考をしても、正しい問いに答えていない可能性がある。

---

## Part 1: 論点とは何か

### 定義

**論点** = 「今、この状況で考えるべき問い」

- 問題全体の中から「本質的に重要な問い」を絞り込んだもの
- 答えることで「状況が動く」問い
- 論点を外した議論は、どれだけ精緻でも価値が出ない

### 論点 vs 問題

| | 問題（Problem） | 論点（Issue） |
|--|---------------|-------------|
| 定義 | 現状と目標のギャップ全体 | そのギャップの中の「本質的な問い」 |
| 数 | 多数ある | 絞り込まれた少数（1〜3個） |
| 特徴 | 「困っていること」を全部含む | 「これを解けば動く」もの |

## Part 2: 論点の4段階構造

論点は以下の階層で構造化される。

```
全体論点（Overall Issue）
    ↓ 分解
サブ論点1  サブ論点2  サブ論点3
    ↓ さらに分解
サブサブ論点 …
```

- **全体論点**: プロジェクト・案件の「問い全体を1文に圧縮したもの」
- **サブ論点**: 全体論点を解くために答えが必要な構成要素の問い
- 論点は「答えると次の論点が見えてくる」形で連鎖する

### 全体論点の設定

```
全体論点の条件:
1. 1文で書ける（長いと絞り込めていない証拠）
2. Yesか Noか、AかBかで答えられる（または一言で答えられる）
3. 答えると「何かが変わる・動く」
4. 今の状況で実際に検討に値する
```

## Part 3: 論点を見つける3ステップ

### Step 1: 論点候補を広げる

まず「問題になりうること」を広く出す。

```
問いかけの例:
- 「この状況で答えが必要な問いは何か？（全て列挙する）」
- 「ステークホルダーが最も気にしていることは何か？」
- 「これを解決しないと何が困るか？」
- 「これを解決すると何が変わるか？」
```

### Step 2: 論点を絞り込む（4つの基準）

広げた論点候補から「本当の論点」を選ぶ。

**① 重要性**
- 「これを解くと、どれほど大きな変化が起きるか？」
- インパクトが大きいほど、論点として価値が高い

**② 緊急性**
- 「今、この状況で答えるべき問いか？」
- 正しくても今関係ない論点は後回しにする

**③ 解決可能性**
- 「今の時間・情報・リソースで答えが出せるか？」
- 答えが出せない論点は論点として機能しない

**④ 依存関係**
- 「この論点の答えが出ないと、他の論点に取り組めないか？」
- 先行必須の論点は優先順位が高い

### Step 3: 論点の構造化（ツリー化）

絞り込んだ全体論点を、サブ論点に分解する。

```
全体論点: 「この新製品は投入すべきか？」

サブ論点の分解例:
  ├─ 「市場に十分な需要があるか？」
  ├─ 「自社に勝てる競合優位性があるか？」
  └─ 「採算が取れるか？」

各サブ論点についてさらに分解:
  「市場に需要があるか？」
    ├─ 「市場規模はどのくらいか？」
    └─ 「顧客が現在の製品に不満を持っているか？」
```

## Part 4: 論点の「ずれ」を避ける

論点思考の最大のリスクは「依頼された問い」と「本当の論点」がずれていること。

### 論点のずれの3パターン

**① 広すぎる論点**
```
依頼: 「会社の業績を改善するにはどうすればよいか」
問題: 広すぎて答えが出ない

修正: 「今期の利益率低下の主因はコストか収益か」
      → 答えが出て、次の行動が見える
```

**② 狭すぎる論点（枝葉に入りすぎ）**
```
依頼: 「A支店の来月の販促チラシの最適なデザインは何か」
問題: 本当の論点は「A支店の集客低下の原因」かもしれない

修正: 「A支店の集客低下の根本原因は何か」から始める
```

**③ 依頼主の思い込みに乗ってしまう**
```
依頼: 「コストを削減するためにどの部門を縮小すべきか」
前提: 「コスト削減が唯一の解決策」という思い込みが埋め込まれている

修正: 「業績改善のためにコスト削減と収益改善、どちらに注力すべきか」
      → 前提を疑い、論点を上位に持ち上げる
```

### 論点のずれチェック

```
[ ] 依頼された問いの「前提」を疑ったか？
[ ] 「なぜこの問いが今問われているか」を確認したか？
[ ] より上位の論点（Why?）を考えたか？
[ ] この問いに答えても状況が変わらない可能性はないか？
```

## Part 5: 論点を固定して深める

論点が定まったら、それを「固定」してから取り組む。

### 論点が途中で動くと何が起きるか

- 分析・議論が蓄積されない（全部やり直し）
- チームの方向性がバラバラになる
- 「たくさん考えたが何も決まっていない」状態になる

### 論点の更新は「意図的に」行う

論点は途中で変わってもよいが、**意図的に変更し、全員で共有する**。

```
論点の更新フロー:
  現在の論点で分析 → 
  「この論点は本当に重要か？」と問い直す →
  より本質的な論点が見えた → 
  チームで合意して論点を更新 → 
  新しい論点で分析再開
```

## Part 6: 論点思考 × イシューからはじめよ の関係

| 概念 | 論点思考（内田） | イシューからはじめよ（安宅） |
|------|---------------|--------------------------|
| 問いの選定 | 論点の設定・絞り込み | イシューの見極め |
| 問いの分解 | サブ論点への構造化 | イシューの分解 |
| 答えの方向 | 仮説思考へ渡す | 仮説→解→伝える |
| 価値の源泉 | 正しい問いを設定する | イシュー度 × 解の質 |

**統合すると**:
```
論点思考: 「何を問うか」を決める
イシュー  : 「その問いは本当に重要か」を判断する
仮説思考: 「その問いにどう答えるか」の仮説を立てる
```

## 実践チェックリスト

### 作業・分析を始める前に

```
[ ] 今の「論点（問い）」を1文で書けるか？
[ ] その論点は「依頼された問い」と同じか？（ずれていないか）
[ ] 論点の前提を疑ったか？（より上位の問いはないか）
[ ] 論点に答えると何が変わるか（重要性）を確認したか？
[ ] 今の時間・リソースで答えが出せる論点か（解決可能性）？
```

### 論点を構造化するとき

```
[ ] 全体論点が1文で書けているか？
[ ] サブ論点はMECEになっているか？
[ ] サブ論点を全部解くと全体論点に答えられるか？
```

## Anti-patterns

| パターン | 問題 |
|---------|------|
| **論点を設定せずに分析開始** | 何のために分析しているか不明になる |
| **依頼された問いをそのまま論点にする** | 依頼主の思い込みに乗ってしまう |
| **論点が広すぎる** | 「業績改善は？」→ 答えが出ない |
| **論点が狭すぎる** | 枝葉の問いを解いても本質が動かない |
| **論点が途中で無意識にずれる** | 気づかないうちに別の問いを解いている |
| **複数の論点を同時に追う** | 焦点が分散して深く入れない |
| **論点の更新を恐れる** | 間違った論点に固執して時間を浪費する |
