# システム思考 — 思考フレームワーク

> 出典: ドネラ・H・メドウズ『世界はシステムで動く』（Thinking in Systems: A Primer）  
> 著者: Donella H. Meadows（マサチューセッツ工科大学 システムダイナミクス研究者）  
> 用途: 複雑な問題・反直観的な現象・長期的変化を理解するための根本フレームワーク

---

## 0. 根本原則

```
問題の原因は「システムの外」ではなく「システムの構造」の中にある。
要素を変えるより、構造を変えよ。
```

なぜシステム思考が必要か:
- **短期的な介入が長期的な悪化を招く**（例: バグを直すたびに別のバグが生まれる）
- **見えない原因が見えない場所にある**（原因と結果が時間的・空間的に離れている）
- **善意の介入が逆効果になる**（「症状」を抑えると「根本原因」が悪化する）

---

## Part 1: システムの3要素

全てのシステムはこの3要素から構成される。

### ストック（Stock）
- システム内に**蓄積・貯留**されるもの
- ある時点の「量」として計測できる
- 変化はゆっくり（突然増減しない）
- 例: 水槽の水量、人口、信頼感、知識、資金残高、在庫

### フロー（Flow）
- ストックを**変化させる速度**
- 単位時間あたりの量（率）として計測する
- 例: 降水量・蒸発量（水槽）、出生率・死亡率（人口）、学習速度（知識）

```
        流入フロー
            ↓
[  ストック  ] ──→ 流出フロー
```

### フィードバックループ
- ストックの変化が、次のフローに影響する**循環**
- フィードバックなしにシステムは成立しない

---

## Part 2: フィードバックループの2種類

### 強化ループ（Reinforcing Loop / R ループ）

```
変化が同じ方向に増幅される（正のフィードバック）
```

- ストックの増加 → フローが増加 → さらにストックが増加
- または: ストックの減少 → フローが減少 → さらにストックが減少
- **指数的成長または崩壊**をもたらす

```
具体例:
  口コミ: ユーザー数↑ → 口コミ増↑ → ユーザー数↑（→ネットワーク効果）
  学習:   知識↑ → 理解速度↑ → 知識↑（→ 習熟の加速）
  貧困:   貧困↑ → 教育機会↓ → 貧困↑（→ 貧困の連鎖）
  技術的負債: バグ↑ → 開発速度↓ → 回避コード↑ → バグ↑
```

強化ループを見つける問い:
- 「この変化は自分自身をどう増幅するか？」
- 「成功がさらなる成功を呼んでいる仕組みはないか？」

### バランシングループ（Balancing Loop / B ループ）

```
目標値に向かって調整される（負のフィードバック）
```

- ストックと目標値のギャップを縮めようとする
- 安定・定常状態・自己修正を生み出す
- ループ内に**遅延（Delay）**があると振動や過剰補正が起きる

```
具体例:
  サーモスタット: 温度↑ → 暖房OFF → 温度↓ → 暖房ON（→ 目標温度を維持）
  在庫管理: 在庫↓ → 発注↑ → 在庫↑（→ 目標在庫を維持）
  体温調節: 体温↑ → 発汗↑ → 体温↓
```

バランシングループを見つける問い:
- 「何かが目標値に向かって修正されているか？」
- 「ここで達成しようとしている目標（明示・暗示）は何か？」

## Part 3: 遅延（Delay）

フィードバックループの中で、原因と結果の間にある**時間差**。

### 遅延がシステムに与える影響

```
遅延なし: 安定した調整
遅延あり: 振動（行き過ぎ → 戻しすぎ → 行き過ぎ）
遅延が大きい: 発散（制御不能）
```

**シャワーの比喩（ミドウズの説明）**:
- 水温を上げる → 熱くなるまで10秒かかる → 待ちきれずさらに熱くする → 熱すぎた → 冷たくする → 冷たすぎた → 繰り返し
- 解決策: 遅延を意識して「小さく動かして待つ」

### 遅延の実例

| 状況 | 遅延 | 振動の例 |
|------|------|---------|
| 採用・教育 | 数ヶ月〜年 | 人員過剰→採用停止→人員不足→大量採用 |
| 政策効果 | 数年 | 規制→市場反応→規制緩和→再規制 |
| 学習・研究 | 年単位 | 投資→無成果→撤退→再参入 |
| MLモデルのフィードバック | バッチサイクル | 過学習→再学習→別方向の過学習 |

## Part 4: システムの8つの罠

複雑なシステムが繰り返し陥る問題パターン。パターンを認識すれば対処できる。

### 1. 政策抵抗（Policy Resistance）

**構造**: 複数のアクターがそれぞれの目標を持ち、介入するたびに他のアクターが補正行動を取る  
**例**: 薬物取締を強化するたびに密売が地下化・高度化する  
**脱出策**: 全アクターの目標を統合するシステム設計。強制ではなく協調

### 2. 公有地の悲劇（Tragedy of the Commons）

**構造**: 共有資源を個人が利用し、利益は個人に・コストは全体に分散される強化ループ  
**例**: 共有リソースの枯渇、過剰採漁、技術的負債の蓄積  
**脱出策**: 利用量の可視化・制限・課金。共有資源の管理責任の明確化

### 3. 目標のドリフト（Drift to Low Performance）

**構造**: 目標が達成困難なとき、目標自体を下げることで「問題解決」とする  
**例**: 品質基準を達成できないので基準を緩める。目標を下げ続ける  
**脱出策**: 目標を固定し、絶対基準として守る。過去の最良値を参照点にする

### 4. エスカレーション（Escalation）

**構造**: 一方の強化が他方の強化を引き起こす対立的な強化ループ  
**例**: 軍拡競争、価格競争、報復合戦  
**脱出策**: 一方が一方的に緩める。上限設定の合意。ループから降りる

### 5. 成功者がさらに成功する（Success to the Successful）

**構造**: 成功したアクターにさらにリソースが集まる強化ループ  
**例**: 富の集中、勝者総取り市場、先行者優位  
**脱出策**: 格差を修正する仕組み（累進課税・再分配・多様性確保）

### 6. 依存と侵食（Shifting the Burden）

**構造**: 症状への対処が根本原因への取り組みを侵食し、依存を生む  
**例**: 痛み止めで痛みを抑えて根本治療をしない。応急処置が定常化する  
**脱出策**: 根本原因のソリューションを優先。症状対処は一時的であると明示

### 7. ルール破り（Rule Beating）

**構造**: ルールの文字を守りながら精神を破る行動が誘発される  
**例**: KPI達成のためのデータ操作。規制の抜け穴探し  
**脱出策**: ルールの目的（精神）を明示する。KPIを目標と混同しない

### 8. 目的の誤り（Seeking the Wrong Goal）

**構造**: 真の目標の代わりに代理指標を最適化し、本来の目的が失われる  
**例**: GDPが高いが国民が幸せでない。精度が高いがビジネス価値がない  
**脱出策**: 真の目標を問い直す。代理指標を定期的に見直す

## Part 5: レバレッジポイント（12箇所）

システムを変えるための介入ポイント。**下（12番）ほど効果が低く、上（1番）ほど効果が高い**。

| # | レバレッジポイント | 説明 |
|---|----------------|------|
| 12 | **数値・定数・パラメータ** | 税率・速度制限・補助金額。最も変えやすいが効果最小 |
| 11 | **バッファのサイズ** | 在庫・余力・予備。安定性に影響するが変えにくい |
| 10 | **ストック・フロー構造** | 物理的インフラ。変えると効果大だが困難 |
| 9 | **遅延の長さ** | 重要な遅延の短縮・延長。状況による |
| 8 | **バランシングループの強さ** | 負のフィードバックの調整力 |
| 7 | **強化ループの強さ** | 成長ループの勢い |
| 6 | **情報フロー構造** | 誰に何がどのタイミングで見えるか。**比較的変えやすく効果大** |
| 5 | **ルール** | インセンティブ・法律・制度 |
| 4 | **自己組織化の力** | システムが自らの構造を変える能力 |
| 3 | **目標** | システムが目指すもの。**非常に強力** |
| 2 | **パラダイム** | システムが生まれる前提・世界観・信念 |
| 1 | **パラダイムを超える** | 全てのパラダイムはモデルに過ぎないと気づく。最高のレバレッジ |

### 実践的含意

```
よくやる介入: 12（パラメータ変更）→ 効果薄
本当に効く介入: 6（情報フロー）+ 3（目標の再設定）+ 2（前提の見直し）

「頑張っているのに変わらない」→ パラメータ介入をしているだけ
「劇的に変わった」→ 目標・パラダイムが変わっている
```

## Part 6: システム思考の実践プロセス

```
Step 1: 何が「ストック」か特定する
  → 時間的に蓄積・減少するものは何か？

Step 2: フローを特定する
  → ストックを増やす・減らすものは何か？

Step 3: フィードバックループを描く
  → 強化ループ（R）かバランシングループ（B）か

Step 4: 遅延を探す
  → 原因と結果の間に時間差はあるか？

Step 5: どの罠に陥っているか確認する
  → 8つの罠パターンに当てはまるものはないか？

Step 6: レバレッジポイントを探す
  → 今やっている介入は何番か？より高いレバレッジポイントはどこか？
```

---

## 実践チェックリスト

```
[ ] 「問題の原因」を人・部門・イベントに帰属していないか？
    → 原因はシステムの構造にある場合が多い

[ ] 強化ループとバランシングループを識別できているか？

[ ] 遅延はどこにあるか？どれくらいの長さか？

[ ] 「症状への対処」をしているか？「根本原因への対処」をしているか？

[ ] 今の介入はレバレッジポイントの何番か？
    → もっと高いレバレッジポイントに移れないか？

[ ] 「8つの罠」のどれかに当てはまっていないか？
```

## Anti-patterns

| パターン | 問題 |
|---------|------|
| **原因を人に帰属する** | 「あの人が悪い」→ 構造が変わらないと同じ問題が繰り返す |
| **短期修正の繰り返し** | 症状を抑えるたびに根本原因が悪化する（Shifting the Burden） |
| **パラメータだけ変える** | 税率・予算・KPI数値を変えても構造が変わらない |
| **遅延の無視** | 「介入したのに効果がない」→ 遅延が来ていないだけ |
| **ループの方向を見誤る** | 強化ループだと思っていたものが実はバランシングループ |
| **目標の代理指標化** | 代理指標を最適化して本来の目標を失う（Seeking the Wrong Goal） |
