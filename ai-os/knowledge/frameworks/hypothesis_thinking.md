# 仮説思考 — 思考フレームワーク

> 出典: 内田和成『仮説思考 BCG流 問題発見・解決の発想法』（東洋経済新報社）  
> 著者: 内田和成（ボストン コンサルティング グループ 日本代表）  
> 用途: 情報が少ない段階から答えに向かって進むあらゆる思考・分析の基盤

---

## 0. 根本原則

```
仮説なき情報収集は、無限に続く砂漠の旅と同じ。
仮説があれば、最短距離で答えに辿り着ける。
```

- **仮説思考** = 今持っている情報を元に「最善の答え（仮説）」を出し、それを検証・更新する思考法
- 完璧な情報を揃えてから考えるのではなく、**不完全な情報でも答えを出す力**
- BCGの現場では「プロジェクト開始初日から仮説を持つ」ことが求められる

### 仮説思考 vs 情報収集型思考

| | 情報収集型 | 仮説思考型 |
|--|-----------|-----------|
| 出発点 | 情報・データ | 「答え」の仮説 |
| 情報収集 | 網羅的・全量 | 仮説検証に必要な情報だけ |
| 思考の向き | 帰納（事実→結論） | 演繹+修正（仮説→検証→更新） |
| スピード | 遅い | 速い |
| リスク | 確証バイアスに気づきにくい | 仮説が崩れることがある（それが正常） |

## Step 1: 仮説を立てる

### 仮説の2種類

**① 課題設定仮説（What's the issue?）**
- 「何が本当の問題か」についての仮説
- 表面的な現象ではなく、その背後にある本質的な原因を特定する
- 例: 「売上が落ちているのは、製品力ではなく営業プロセスの問題ではないか」

**② 解決策仮説（What's the solution?）**
- 「どう解決するか」についての仮説
- 課題設定仮説が固まってから立てる（順番が重要）
- 例: 「営業プロセスを見直せば、同じリソースで30%売上が上がるのではないか」

### 仮説の質を高める問いかけ

```
1. 今持っている情報で「答えらしきもの」を一言で言えるか？
2. その答えが正しければ、何が説明できるか？
3. 他に考えられる答えを3つ出せるか？（最初の仮説に固執しない）
4. この仮説を否定するデータがあるとしたら何か？
```

### ABC アプローチ

```
Always Be Creating hypotheses.
（常に仮説を作り続ける）
```

情報に接するたびに「この情報は自分の仮説を支持するか、否定するか」を自問する。

## Step 2: 空・雨・傘

仮説思考の最も基本的なフレームワーク。事実から行動までを一本の論理でつなぐ。

```
空（事実・現象）
    ↓ 解釈・洞察（ここが仮説）
雨（解釈・課題認識）
    ↓ 意思決定
傘（行動・提言）
```

| レベル | 問い | 例 |
|-------|------|---|
| **空** | 何が起きているか（事実） | 「Q1の営業成績が全員20%下落した」 |
| **雨** | なぜ起きているか（仮説） | 「新製品への切り替え期で説明が追いついていないのではないか」 |
| **傘** | だからどうするか（行動） | 「新製品のトークスクリプトと同行訪問を今月中に実施する」 |

### よくある失敗

- **空だけ語る**: 「売上が落ちています」（で？という状態）
- **雨を飛ばす**: 空→傘に直行（なぜその行動か説明できない）
- **傘の傘**: 行動を行動で説明する（「売上が落ちたので営業を強化します」）

## Step 3: 仮説を検証する

### 検証の順序

```
1. キークエスチョンを特定する
   → 「この仮説が正しいなら、必ず観察されるはずのことは何か？」

2. 反証から入る
   → 仮説を支持する情報より、否定する情報を先に探す

3. 最小の情報で最大の検証をする
   → 「これを見れば答えが出る」という1点を見つける

4. 仮説を更新する
   → 崩れたら修正する。崩れること自体は失敗ではない
```

### キークエスチョンの作り方

```
仮説: 「〇〇が原因ではないか」

キークエスチョン:
  - 仮説が正しければ → [何が観察されるはず？]
  - 仮説が間違っていれば → [何が観察されるはず？]

観察した結果:
  → 仮説を支持する / 否定する
```

### 仮説更新のループ

```
初期仮説
    ↓
検証設計（何を見れば確認できるか）
    ↓
情報収集・分析（最小限）
    ↓
仮説を支持 → 次のサブ仮説へ
仮説を否定 → 代替仮説へ更新
    ↓
最終仮説（= 答え）
```

## Step 4: 仮説思考の深化

### 「なぜ？」を3回繰り返す

```
現象: 「顧客が離脱している」
Why 1: 「なぜ？」 → 「製品に満足していないから」
Why 2: 「なぜ？」 → 「サポートレスポンスが遅いから」
Why 3: 「なぜ？」 → 「サポートチームの人手が足りないから」

課題設定仮説: 「本質的な問題はサポート体制の人員不足ではないか」
```

3回で届かない場合は5回。ただし「なぜ」を繰り返すだけでなく、**別の原因ルートも並行して考える**。

### 仮説の深さ vs 広さ

- **広い仮説**: 「売上問題の原因は①製品②営業③マーケの3つのどれか」（MECEに構造化）
- **深い仮説**: 「営業プロセスの中でも、クロージング段階に問題がある」（特定化）

実務では**広さで網羅 → 深さで絞る**の順。

## 仮説思考の実践チェックリスト

### 何かを始める前に

```
[ ] 今の段階で「答えの仮説」を一言で言えるか？
[ ] 課題設定仮説（何が問題か）と解決策仮説（どう解くか）を区別しているか？
[ ] 空（事実）・雨（解釈）・傘（行動）を分けて書けるか？
```

### 情報収集・分析中に

```
[ ] この分析は仮説のどこを検証しているか？
[ ] 仮説を否定するデータを積極的に探したか？
[ ] 「全データを集めてから考える」モードに入っていないか？
```

### 結論を出す前に

```
[ ] 仮説は最後まで検証されたか（された → 採用、否定された → 更新）？
[ ] 空・雨・傘が一本の論理でつながっているか？
[ ] 「どうしてそう言えるの？」に答えられるか？
```

## Anti-patterns

| パターン | 問題 |
|---------|------|
| **情報収集地獄** | 仮説なしに全情報を集めようとする。終わらない |
| **確証バイアス** | 仮説を支持する情報だけ見る。否定情報を無視 |
| **仮説を変えることへの恐れ** | 仮説が崩れることは失敗ではなく、前進 |
| **雨のない空と傘** | 事実→行動で「なぜその行動か」の論拠がない |
| **深さなき広さ** | 「3つの原因が考えられます」で止まり、どれかに絞らない |
| **精緻な仮説の先送り** | 「もう少し情報が集まってから」と仮説を出さない |
