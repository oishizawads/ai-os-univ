## Summary
特徴量エンジニアリングは「何でも作る」ではなく「仮説から作る」。
「このデータのどの構造をモデルに教えたいか」を先に言語化する。
リークしやすい特徴量パターンを知っておくことが最も重要。

## 特徴量の種類と使いどき

### 数値変換
| 変換 | 使うとき |
|---|---|
| Log変換 | 右裾が重い分布（売上・アクセス数等） |
| Box-Cox | 正規化が必要なとき（線形モデル向け） |
| 二乗・平方根 | 非線形関係が疑われるとき |
| Binning | 数値をカテゴリ的に扱いたいとき |

### 集計特徴量
| 特徴量 | 使うとき | リーク注意 |
|---|---|---|
| グループ集計（mean/std/max） | グループ内の位置を表現 | fold外で計算するとリーク |
| Target encoding | カーディナリティの高いカテゴリ | 必ずfold内でfit |
| Count encoding | 出現頻度が重要なとき | testに出現しないカテゴリに注意 |
| Rank encoding | 外れ値に強くしたいとき | - |

### 時系列特徴量
| 特徴量 | 使うとき | リーク注意 |
|---|---|---|
| Lag特徴量 | 過去の値が予測に使える | lagが小さすぎると未来が混入 |
| Rolling統計 | トレンド・季節性の平滑化 | 予測時点より未来を使わない |
| Diff特徴量 | トレンド除去・変化率 | - |
| 時刻分解（曜日・時間帯） | 周期性がある場合 | - |
| 累積統計 | 蓄積値が重要な場合 | 未来の蓄積を使わない |

### 交互作用特徴量
- 2特徴量の積・比・差
- ドメイン知識から「この2つの組み合わせが意味を持つ」が言える場合のみ
- 総当たりは過学習リスクが高い → 仮説ベースで選ぶ

## 特徴量設計の手順

1. **ターゲットとの相関構造を仮説化する**
   - 「〇〇が高いほどターゲットが高い/低い」を仮説として書く

2. **生データにある情報を列挙する**
   - 数値・カテゴリ・テキスト・時刻・位置情報

3. **仮説を特徴量に変換する**
   - 仮説ごとに「どの特徴量で表現できるか」を対応させる

4. **リークチェックを通す**（後述）

5. **重要度・有効性を検証する**
   - permutation importance / SHAP で効いているか確認
   - CVスコアが上がらない特徴量は除外を検討

## リーク頻出パターン

### Target leakage
- ターゲットの集計値をfold外で計算する（target encoding）
- ターゲットと相関が高すぎる特徴量（0.99以上は疑う）

### Temporal leakage
- 時系列データで予測時点より未来の情報を使う
- rolling窓が予測時点をまたいでいる
- testセット作成時刻のデータがtrainに含まれている

### Row duplication leakage
- data augmentation後に同一サンプルが異なるfoldに入る
- 重複行がtrainとtestに分かれている

### Preprocessing leakage
- ScalerをOOF全体に対してfitする（test情報が混入）
- 欠損補完の平均値をtestを含めて計算する

## 特徴量選択の原則
- 重要度が低い特徴量を削除 → モデルの汎化性能が上がる場合がある
- 相関が高い特徴量のペアは一方を残す
- 欠損率 > 90% の特徴量は通常不要
- Tree系では特徴量数が多くても比較的ロバスト（線形モデルは削除推奨）

## Eval
- 全ての集計・エンコーディングがfold内で計算されているか
- 時系列特徴量に未来情報の混入がないか
- 特徴量の追加にドメイン仮説があるか
- 追加前後でCVスコアを比較したか
