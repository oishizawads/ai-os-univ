## Summary
Claudeへの指示は「自由作文させない」が基本。
構造・出力形式・判断基準を事前に与えるほど品質が上がる。
「賢さ」に頼るのではなく「型」に落とし込む。

## 効果の高い指示パターン

### 1. 役割を与える
```
あなたは〇〇の専門家です。
〇〇の観点から、〇〇を行ってください。
```
→ 出力のトーン・視点・深さが変わる。「汎用アシスタント」のままにしない。

### 2. 出力形式を固定する
```
以下の形式で出力してください：

## 結論
（1文）

## 根拠
- （箇条書き）

## リスク
- （箇条書き）
```
→ フォーマットを指定しないと出力がばらつく。特にエージェント・パイプラインでは必須。

### 3. 判断基準を明示する
```
以下の基準で判断してください：
- 〇〇の場合は「A」
- 〇〇の場合は「B」
- どちらでもない場合は「C（理由付き）」
```
→ Claudeに判断を委ねすぎない。決定ロジックを明示する。

### 4. やらないことを指定する
```
以下はやらないでください：
- 一般論・教科書的な説明
- 〇〇については触れない
- 推測の場合でも断定しない
```
→ 「やること」より「やらないこと」の方が制約として効くことが多い。

### 5. 長さ・粒度を指定する
```
回答は3文以内で。
各項目は50文字以内で。
詳細は不要。要点のみ。
```
→ 長い回答を求めていないときは明示する。

## システムプロンプト vs ユーザープロンプト

| 種別 | 入れるもの |
|---|---|
| System prompt | 役割・制約・出力形式・禁止事項（変わらないルール） |
| User prompt | タスク・データ・コンテキスト（実行ごとに変わる内容） |

ルールをUser promptに書いてしまうと、長い入力で忘れやすくなる。

## Chain-of-Thoughtの使いどき

```
# 有効な場合
- 複雑な推論・数値計算・多段階の判断
- 「まず〇〇を考えてから〇〇をしてください」と段階を明示

# 不要な場合
- 単純な分類・抽出タスク
- 出力形式が固定されている場合（CoTがフォーマットを崩す）
```

## Few-shotの使いどき
- 出力フォーマットが複雑なとき（例を見せる方が指示より正確）
- 判断の粒度を揃えたいとき（例で「このくらいの詳細度」を示す）
- Zero-shotで出力品質が安定しないとき

```
# 例の書き方
入力: [具体例]
出力: [期待する出力]

入力: [別の具体例]
出力: [期待する出力]

入力: [実際のタスク]
出力:
```

## 長いコンテキストを渡すときの注意
- 重要な情報は最初か最後に置く（中間は見落とされやすい）
- 「以下の〇件のドキュメントから〇〇してください」と件数を明示
- 不要な情報は渡さない（プロンプトを短くする方が精度が上がることが多い）

## このシステムへの適用

knowledge-pipelineのプロンプトで使っているパターン:
- System promptで役割・出力規則を固定（`REPORT_SYSTEM_PROMPT`等）
- User promptでクエリとドキュメントを渡す
- 出力形式をUser prompt内でMarkdownテンプレートとして明示

## Anti-patterns
- 「うまくやってください」「ベストを尽くして」のような抽象的な指示
- 出力形式を指定せずに「自然な形で」と言う
- 1つのプロンプトに複数の異なるタスクを詰め込む
- 制約を後から追加していく（設計段階で決める）
- 結果が悪かったときにプロンプトだけ直す（入力データの問題かもしれない）

## Eval
- 出力フォーマットが固定されているか
- 役割・制約が system prompt にあるか
- 「やらないこと」が指定されているか
- Few-shotが必要なタスクに例があるか
