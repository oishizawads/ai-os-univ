## Summary
技術的に正しい内容でも、伝わらなければ意思決定に使われない。
「何が言いたいか」を最初に言う。理由・根拠は後から。
相手が知りたいのは「で、どうするの？」だけ。

## ピラミッド原則

構造: **結論 → 根拠 → 詳細**

```
結論（最初に言う）
  ├── 根拠1
  │     └── 詳細・データ
  ├── 根拠2
  │     └── 詳細・データ
  └── 根拠3
        └── 詳細・データ
```

### よくある逆ピラミッドの失敗
```
分析内容の説明 → 手法の説明 → 結果の数字 → 「つまり...」
```
→ 聞き手は最後まで「で、何が言いたいの？」と思っている

## So What? フレーム

全ての分析アウトプットに対して問う：
- **So What?** この結果から何が言える？
- **Now What?** 次に何をすべきか？
- **Why So?** なぜそうなっているか？

数字を出して終わりにしない。数字の「意味」を言語化する。

## 相手別の伝え方

### 意思決定者（経営層・クライアント上位）
- 結論と推奨アクションだけ最初に
- 数字は比較・差分で伝える（「精度0.82」より「現状比+15%」）
- 不確実性を隠さない（「確実に〇〇です」は信頼を失う）
- 詳細を聞かれてから掘り下げる

### 担当者・現場
- プロセスと判断根拠を共有する
- 自分でも検証できるようにする
- 制約・前提を明示する

### 技術者
- 仮定・制限・再現方法を明記する
- 数値は生データで出す（丸めない）

## 報告書・スライドの構成

```
1. エグゼクティブサマリー（結論・推奨・重要な数字のみ）
2. 背景・問題設定（なぜこれをやったか）
3. 分析内容（根拠）
4. 結果（数字と解釈）
5. 推奨アクション（具体的に誰が何をするか）
6. 付録（詳細・技術的内容）
```

## 不確実性の伝え方
- 「精度は〇〇%です」ではなく「精度は〇〇%（±〇〇、95%信頼区間）です」
- 「この予測は外れることがあります」ではなく「この予測が外れやすい条件は〇〇です」
- 「データが不足しています」ではなく「〇〇のデータが追加されれば精度が△△%改善すると推定されます」

## Anti-patterns
- 分析プロセスを全部説明してから結論を言う
- 「結論はどちらとも言えます」で締める（意思決定の役に立たない）
- 技術用語を説明なく使う
- 数字を羅列して解釈を書かない
- 推奨アクションを曖昧にする（「検討が必要です」）

## Eval
- 最初の1文で結論が言えているか
- 「So What?」に答えられているか
- 推奨アクションが具体的か（誰が・何を・いつまでに）
- 不確実性が正直に伝わっているか
