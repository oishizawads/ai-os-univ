## Summary
Vibe Codingとは「自然言語90%以上でプログラムを作成し、人はほぼコーディングしない状態」。
本質はプロンプトの技術ではなく、**AIが100%のパフォーマンスを出せる問題設定（枠組み作り）**にある。
PoC・競技・一人作業に向く。チーム同期開発・クライアント案件は `client_delivery.md` を優先する。

## 前提条件
- プログラミングの基礎力（メモリ管理・データ構造・デザインパターン）が必須
- AIの出力を正しくレビューできるシニアレベルの判断力が必要
- ジュニアが使うと品質低下・スキル停滞のリスクが高い

## AIが最大パフォーマンスを出す4条件

| 条件 | 内容 |
|------|------|
| 情報量の制限 | 1回にAIが考えるコンテキストを極力減らす |
| 連携の排除 | 作業を完全に独立させ、システム間連携を考えさせない |
| サンプルの提示 | Few-shotで成功率を上げる |
| 一般的な「型」の利用 | AIが大量に学習済みの構造・パターンを指定する |

## 3つの実装手法

### 1. 徹底的なモジュール化
- システムを独立したパーツに分割する
- コンポーネント間の連携はファイル単位など最小インターフェースに絞る
- 「今作っている機能だけに集中できる」状態を設計で作る
- AIには「今から作る機能に関係する設計書」だけを渡す

### 2. デザインパターンの活用
- Factory / Template / Strategy などのパターンを明示指定する
- パターンを指定すると自然にFew-shotサンプルが生まれ、AIの出力精度が上がる
- 「〇〇パターンで書いて」だけでAIは構造を理解するのでコミュニケーションコストが下がる

### 3. 階層的設計書の作成
AIのコンテキスト負荷を下げるために4層に分割する：

```
概念設計書   → アーキテクチャ設計書 → コンポーネント設計書 → 詳細設計書
（Why・What）   （全体構造）           （モジュール単位）       （実装仕様）
```

- 一度に全要件をAIに渡さない
- 実装時はそのモジュールの「詳細設計書」だけを渡す
- 全体仕様書を毎回読ませない

## 作業ループ

```
1. 概念をAIに伝え、分割方法を相談（コードはまだ書かない）
2. コンポーネント設計 → うまくいかなければ概念から再検討
3. 詳細設計 → レビューし、必要ならコンポーネント再設計
4. 実装をAIに任せる
5. テスト → 設計書の修正に戻る
```

「壊して作り直す」サイクルが連続する。これはPoC・競技では正常な状態。
記録より速度を優先する局面では、最低限「なぜ壊したか」だけ残す。

## レビュー戦略

### 複数AIクロスレビュー
- 同じコード・設計書を Claude Code と GitHub Copilot の両方にレビューさせる
- AIによって指摘観点が異なるため、異なる箇所の問題を拾える
- アーキテクチャ変更など広範囲の修正はループを複数回回す

### 人間が必ず見るべき箇所
- テストコード（AIは「それっぽいテスト」を書く。中身を確認しないと意味がない）
- システムのコア設計判断（AIだけに委ねると判断の質が落ちやすい）
- セキュリティ・データ契約に関わる箇所

### ツール使い分けの考え方
- ツールの優劣はモデルアップデートで頻繁に変わる
- 「このツールが正解」に固執しない。定期的に最新状態を確認して判断する

## Anti-patterns
- 抽象的な課題を丸投げする（「良い感じにして」）
- 全体仕様書を毎回渡す
- AIのテストコードを確認しない
- コアの設計判断をAIだけに委ねる
- チーム同期開発でVibe Codingの高速ループを使う（属人性が極限まで上がる）
- ジュニアエンジニアにそのまま適用する

## 適用場面
| 向く | 向かない |
|------|---------|
| PoC・競技・一人作業 | チーム同期開発 |
| 仕様変更が多い初期フェーズ | クライアント案件の本番実装 |
| 探索的な実験 | 説明責任・記録が必要な局面 |

## Eval
- コンポーネントが独立していて、AIに渡すコンテキストが最小化されているか
- 設計書が4層に分かれていて、実装時は詳細設計書だけ渡せるか
- テストコードを人間がレビューしたか
- コア設計判断に自分の判断が入っているか
- クロスレビューを実施したか
