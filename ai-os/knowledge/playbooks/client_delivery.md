## Summary
クライアント案件は「良いものを作る」より「合意した成果を期待通り届ける」が優先。
要件の曖昧さは早期に潰す。後から出てくる要件変更が最大のリスク。
意思決定とその根拠を記録することが説明責任の基盤になる。

## キックオフ前に確認すること

### 必須確認項目
- [ ] 最終成果物は何か（レポート・モデル・システム・提案書）
- [ ] 誰が使うのか（意思決定者・現場担当者・エンドユーザー）
- [ ] 成功基準は何か（KPI・定量指標）
- [ ] 失敗条件は何か（これが出たら撤退・方針転換）
- [ ] データの契約・権利・提供形式
- [ ] スコープのOUT（やらないことの明文化）

### データコントラクト確認
- [ ] データの提供タイミングと形式（CSV・DB・API）
- [ ] カラム定義・ユニット・欠損の扱い
- [ ] 更新頻度と履歴の有無
- [ ] 個人情報・機密情報の取り扱い制約
- [ ] 本番データとサンプルデータの違い

## 要件定義のフォーマット

```
目的: （1文、なぜこのプロジェクトをやるか）
成功条件: （数値で定義。誰が何をもって成功と判断するか）
失敗条件: （数値で定義。何が出たら撤退・代替案を取るか）
スコープIN: （やること）
スコープOUT: （やらないこと）
制約: （技術・データ・期間・コスト・組織）
前提: （このまま動いていて欲しい仮定）
```

## コミュニケーション設計

### 報告頻度の目安
| フェーズ | 頻度 | 内容 |
|---|---|---|
| 要件定義 | 毎回 | 理解の確認・前提の合意 |
| 実装中 | 週1 | 進捗・ブロッカー・認識ズレ確認 |
| 成果確認 | 都度 | 中間アウトプットのレビュー |
| 納品 | 1回 | 最終成果物の説明・引き継ぎ |

### やること
- 毎回「次回までに何をするか」を文書で送る
- 認識が違うと感じたら即確認（後回しにしない）
- 変更要求は必ず影響範囲を見積もって返す

### やらないこと
- 口頭だけで合意する（後で「言った/言わない」になる）
- 認識のズレを放置する
- スコープ外の要求をその場で受ける

## 意思決定ログの管理
重要な決定（技術選択・スコープ変更・方針転換）は必ず記録する。

```
日付:
決定内容:
背景・理由:
却下した選択肢と理由:
影響範囲:
確認者:
```

## 納品チェックリスト
- [ ] 成功条件を満たしているか（定量的に確認）
- [ ] 成果物の形式が合意通りか
- [ ] ドキュメントが揃っているか（手順書・制限事項・前提条件）
- [ ] 引き継ぎに必要な情報が揃っているか
- [ ] 次フェーズの提案・未解決事項のリストがあるか

## Anti-patterns
- 要件が決まる前に実装を始める
- 「たぶんこういう意味だろう」で進める
- 中間レビューなしに最終納品まで進める
- スコープ外要求を断れずに抱え込む
- 失敗した試みを報告しない（後で信頼を失う）

## Eval
- 成功条件が数値で定義されているか
- スコープのIN/OUTが合意されているか
- 変更要求に対して影響範囲を見積もったか
- 意思決定ログが残っているか
