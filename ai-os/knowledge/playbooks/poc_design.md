## Summary
PoCを「作って見せる」から「仮説を検証する」に変える。
成功条件を事前に決め、それを満たすかどうかを問う。

## Procedure

### Phase 1: PoC設計前の確認
1. 何を証明するPoCか（仮説を1文で書く）
2. 誰に何を納得させるPoCか（意思決定者・基準を明確に）
3. 成功条件を数値で定義する
4. 失敗条件も定義する（何が出たら別の方向に進むか）
5. 期間・リソース・データ制約を確認

### Phase 2: スコープ定義
1. IN: PoCで検証することを列挙
2. OUT: PoCで検証しないことを明記
3. 前提条件・依存関係を整理
4. 利用データの確認（本番データか・サイズは十分か）

### Phase 3: 実装計画
1. ベースライン（最もシンプルな解）を先に定義
2. 評価指標と計算方法を実装前に合意
3. タスク分解・担当・期限を決める
4. レビューポイントを設定

### Phase 4: 実施・記録
1. ベースラインを最初に実装して動かす
2. 各実験の目的・結果・解釈を記録
3. 仮説の更新を都度記録
4. 失敗した試みも記録する

### Phase 5: まとめ・提案
1. 仮説の検証結果（成功/失敗/条件付き成功）
2. 本番化に向けた必要条件を整理
3. リスクと軽減策を明記
4. 次フェーズの提案

## Anti-patterns
- 成功条件を決めずに始める
- スコープが広すぎる
- ベースラインを作らずにいきなり複雑な手法に進む
- 失敗した実験を記録しない
- 「動いた」だけで成功とする

## Eval
- 成功条件が数値で定義されているか
- スコープのIN/OUTが明確か
- ベースラインが定義されているか
- 意思決定者が納得できる結果形式か
