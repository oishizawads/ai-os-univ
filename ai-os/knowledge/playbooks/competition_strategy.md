## Summary
コンペは「良いモデルを作る競争」ではなく「限られた時間で最大スコアを出す戦略ゲーム」。
CV設計を固める → Solidなベースラインを確立 → Explosiveを試す、の順序を守る。
Public LBを信頼しすぎないことが上位との差になる。

## タイムライン設計

### 期間別の優先度
| フェーズ | 期間目安 | やること |
|---|---|---|
| 理解 | 最初の10% | タスク・データ・メトリック・先行事例を把握 |
| CV設計 | 最初の15% | CV確定・ベースライン実装・OOF確認 |
| Solid確立 | 〜50% | 特徴量エンジニアリング・ハイパーパラメータ最適化 |
| Explosive | 〜85% | 外部データ・NN・モデルの多様化 |
| アンサンブル | 最後の15% | 多様性のある予測を組み合わせる |
| 提出準備 | 最後の5% | 形式確認・submit管理 |

### 序盤（〜25%）でやること
- [ ] discussionとnotebookを全部読む（上位が使っている手法を把握）
- [ ] CVをデータ構造に合わせて設計する
- [ ] ベースラインのOOFとPublic LBの乖離を確認する
- [ ] Solid戦略とExplosive戦略の候補を書く

### 中盤でやること
- [ ] 特徴量重要度でデータを理解する
- [ ] CVが改善したときのみPublic LBと比較する（CVを主軸に）
- [ ] 失敗した実験も記録する

### 終盤（〜締切）でやること
- [ ] アンサンブル候補を多様性の観点で選ぶ
- [ ] LB probeをしない（過適合リスク）
- [ ] 最終提出2つを「safe版」と「勝負版」に分ける

## Solidな実験の基準
- CVスコアの改善が再現性を持つ
- fold間のスコアばらつきが許容範囲内（±10%程度）
- 実装がシンプルで追試できる
- Public LBとCVの順位相関がある

## Explosiveな実験の基準
- 失敗してもSolidなスコアを損なわない
- 成功したときのアップサイドが大きい
- 先行研究・上位解法で有効性が示唆されている

## アンサンブル戦略

### 多様性の作り方
- 異なるCV分割で学習する
- 異なるモデルアーキテクチャ（LightGBM + NN等）
- 異なる特徴量セット
- 異なるランダムシード（シードアンサンブル）

### ブレンド比率の決め方
1. OOFスコアで重み付け（スコアが高いほど重みを大きく）
2. 相関が低いモデルを優先的に組み合わせる
3. 単純平均をベースラインとして、加重平均と比較する

### やらないこと
- Public LBが上がるようにブレンド比率を調整する（過適合）
- CVスコアが低いモデルをPublic LBだけで採用する

## Public LB vs Private LBのリスク管理
- Public LBはtestの一部しか使っていない → 過適合しやすい
- CV主軸で選んだモデルを提出の柱にする
- Public LBとCVが乖離しているときは原因を探す（リーク疑い・分布シフト）
- 「shake up」（順位の入れ替わり）を前提にSolidな解を保険として持つ

## Anti-patterns
- CV設計が甘いまま特徴量エンジニアリングを進める
- Public LBが上がったことだけで採用する
- 実験記録をつけずに手動でモデルを管理する
- 終盤にCVを変える
- 全員の同意を得てから実験する（スピードが落ちる）

## Eval
- CVがデータ構造を反映しているか
- Solidな実験でベースラインを確立したか
- 実験ログが追跡可能か
- 最終提出の選択理由を説明できるか
