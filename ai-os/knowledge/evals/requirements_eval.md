## Summary
要件定義・依頼の品質を確認するチェックリスト。
AIへの依頼前、または生成された要件定義のレビュー時に使う。

## 合格条件

### 必須（1つでも欠けたら失格）
- [ ] 問題設定が施策より先にある
- [ ] KPIが数値で定義されている（「改善する」ではなく「〇〇を10%下げる」）
- [ ] 成功条件と失敗条件が両方ある
- [ ] スコープのINとOUTが明示されている
- [ ] 誰が使うか（利用者・意思決定者）が書かれている

### 推奨
- [ ] 制約（期間・データ・コスト・組織）が列挙されている
- [ ] 前提条件が書かれている
- [ ] 不明点が `assumptions.md` に残されている

## 失格パターン

| パターン | 見分け方 |
|---------|---------|
| 一般論から始まる | 「一般的に〇〇は重要で...」で始まる |
| 施策リストが先に来る | 問題設定なしに「〇〇をやる」が並ぶ |
| KPI未定義 | 「精度を上げる」「効率化する」だけで数値がない |
| スコープが曖昧 | やらないことが書かれていない |
| 利用者が「ステークホルダー」のまま | 実際の人物・役割が特定されていない |

## 回帰ケース

### OK例
```
目的: 市区町村×サービス×月次の手配難リスクを数値化し、営業判断に使う
KPI: 手配難発生率の予測精度（F1 ≥ 0.7）
スコープIN: リスクスコアリングモデルの構築・可視化
スコープOUT: リアルタイム配車最適化、業者評価
利用者: 営業マネージャー（月次レビュー用）
```

### NG例
```
目的: AIを使って業務を効率化する
→ 問題設定なし・KPI未定義・スコープ不明
```

## Eval適用タイミング
- クライアントから要件を受け取ったとき
- AIに要件定義を生成させたあと
- PoC設計の前
