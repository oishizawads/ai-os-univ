# Eval Policy

生成物の品質を感覚ではなく基準で管理する。
eval は最後に足すのではなく、タスク開始時に定義する。

---

## 評価の4軸

| 軸 | 問い |
|----|------|
| **Accuracy** | 事実として正しいか。根拠があるか |
| **Usefulness** | 実務で使える粒度か。すぐ行動できるか |
| **Reusability** | 他の案件・次回のセッションでも使えるか |
| **Safety** | リスクを明記しているか。誤用を招かないか |

---

## タスク別評価基準

### 文書・設計
- [ ] 目的に答えているか
- [ ] イシューが明確か
- [ ] 制約・前提が書かれているか
- [ ] 根拠がある（推測は推測と明記している）
- [ ] 次アクションが明確か

### コード
- [ ] 動くか（実行確認済みか）
- [ ] テストを通るか
- [ ] エッジケースに耐えるか
- [ ] 依存関係に問題がないか
- [ ] レビュー観点（`review-exp` or `work-review`）で確認済みか

### 分析・EDA
- [ ] 問題設定が妥当か
- [ ] 評価指標が適切か
- [ ] リーク・データ汚染がないか
- [ ] 再現可能か（seed・データパスが固定されているか）
- [ ] 結論が仮説に答えているか

### 提案・PM
- [ ] KPIに結びついているか
- [ ] 意思決定に使えるか
- [ ] ステークホルダーの懸念に答えているか
- [ ] リスクと軽減策が書かれているか

---

## Regression Check

以下が壊れていないことを確認してからマージ・提出する。

- CVの再現性（コンペ）
- 前処理パイプラインの整合性
- 推論コードの一貫性（train時の変換が inference で再現されているか）

---

## 評価不能な成果物を出さない

以下は必ず拒否・修正する。

- 評価基準が書かれていない提案
- 根拠のない断定
- 「良さそう」だけの判断
- テストなしの本番投入
